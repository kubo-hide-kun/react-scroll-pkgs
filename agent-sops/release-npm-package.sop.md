# Release an npm Package

## Overview

This SOP guides an agent through cutting a new release of one of the
publishable libraries in this Turborepo monorepo (`react-scroll-flip-book`
or `use-window-scroll-in-element`) and publishing it to npm through the
GitHub Actions release workflow.

Use this whenever a package needs a new version on npm: after a bug fix, a
new feature, a dependency change, or a documentation-only update. The steps
encode the recurring release checklist and the failures this project has
actually hit (build breaks failing the whole publish, and out-of-sync
lockfiles breaking `npm ci`) so they do not happen again.

## Parameters

- **package_name** (required): The workspace to release. Must be exactly
  `react-scroll-flip-book` or `use-window-scroll-in-element`.
- **change_summary** (required): A short description of what changed since
  the last release, used to decide the version bump.
- **release_type** (optional, default: "auto"): `patch`, `minor`, `major`,
  or `auto` to derive it from `change_summary` using the criteria in step 2.

**Constraints for parameter acquisition:**

- You MUST confirm `package_name` is one of the two publishable packages
  before proceeding, because `apps/website` is a private demo and MUST NOT
  be published to npm.
- You MUST collect `change_summary` up front so the version bump can be
  justified in a single decision rather than guessed later.

## Steps

### 1. Confirm the working tree and package are release-ready

Verify you are on an up-to-date branch, the target package's source and
`README.md` reflect the change, and there is no uncommitted noise.

**Constraints:**

- You MUST run `git status` and ensure the only pending changes belong to
  this release.
- You MUST NOT hand-edit the compiled output under `lib/`, because it is
  generated by `tsc --project tsconfig.build.json` via `prepublishOnly` and
  any manual edit will be silently overwritten at publish time.
- You SHOULD confirm the package's `files` field still lists only `lib/`
  and `LICENSE`, since anything outside that list will not be published.

### 2. Decide the semantic version bump

Choose `patch`, `minor`, or `major` from `change_summary` using this
project's breaking-change criteria, then bump the version in the package's
`package.json`.

**Constraints:**

- You MUST treat the following as **major**: removing a public API, changing
  a function signature (argument type, order, or requiredness), changing
  default behavior, or a major bump of a runtime dependency.
- You MUST treat backward-compatible feature additions (e.g. new optional
  arguments) as **minor**.
- You MUST treat bug fixes, documentation fixes, and internal refactors with
  no API change as **patch**.
- You MUST NOT reuse or downgrade an already-published version number,
  because npm rejects republishing an existing version and the release will
  fail irrecoverably for that number.
- You SHOULD note the chosen release type and its justification in the
  commit message or PR body.

### 3. Sync dependencies and the lockfile

If the change added, removed, or updated any dependency, install so that
`package-lock.json` is regenerated, then stage the lockfile alongside the
`package.json` change.

**Constraints:**

- You MUST run `npm install` (or `npm install -w <package_name>`) after any
  dependency change and commit the resulting `package-lock.json`.
- You MUST NOT leave `package-lock.json` out of sync with `package.json`,
  because the release workflow runs `npm ci`, which fails hard on any
  lockfile mismatch — this repo has repeatedly broken clean installs and a
  build for exactly this reason.
- You MUST NOT touch `package-lock.json` when no dependency actually
  changed, because a spurious lockfile diff creates review noise and can
  mask a real dependency drift.

### 4. Verify the build and checks are green locally

Run the full pipeline before publishing so a broken build is caught here and
not inside the release workflow.

**Constraints:**

- You MUST run `npm run build` (`turbo run build`) and confirm it succeeds
  for the whole workspace, because the release workflow runs
  `npm run build` before `npm publish`, and a build failure aborts the
  entire publish — this has broken releases in this repo before.
- You MUST run `npm run typecheck` and `npm run lint` and confirm both pass.
- You SHOULD inspect the generated `lib/` (including
  `lib/typescript/index.d.ts`) to confirm the entry point and type
  definitions were emitted.
- You MUST report the result of build, typecheck, and lint as a short
  bulleted list.

### 5. Commit, push, and trigger the publish workflow

Commit the version bump with a Conventional Commit message, push the branch,
and publish through GitHub Actions.

**Constraints:**

- You MUST use a Conventional Commit message with a non-empty scope, e.g.
  `chore(react-scroll-flip-book): release v1.0.3`, because commitlint
  rejects messages without a scope.
- You MUST publish via the GitHub Actions `workflow_dispatch` on
  `publish-component.yml` (for `react-scroll-flip-book`) or
  `publish-hooks.yml` (for `use-window-scroll-in-element`); the workflow runs
  `npm ci` → `npm run build` → `npm publish -w <package_name>`.
- You SHOULD NOT run `npm publish` from a local machine, because the pinned
  CI environment (Node 24, clean install) is the source of truth for what
  ships and local state can differ.
- You MUST verify the new version appears on npm after the workflow
  completes.

## Examples

### Example Input

```
package_name: use-window-scroll-in-element
change_summary: Fixed a throttle edge case; no API change.
release_type: auto
```

### Example Output

- Detected `patch` → bump `1.1.1` → `1.1.2`.
- No dependency change, `package-lock.json` untouched.
- `build`, `typecheck`, `lint` all green; `lib/` and `.d.ts` regenerated.
- Commit `chore(use-window-scroll-in-element): release v1.1.2`, push branch.
- Trigger `publish-hooks.yml` via workflow_dispatch; confirm `1.1.2` on npm.

## Troubleshooting

### The release workflow fails on `npm ci`

`package-lock.json` is out of sync with `package.json`. Run `npm install`
locally, commit the regenerated lockfile, and re-run the workflow.

### The release workflow fails during build

A build error aborts publish before anything ships. Reproduce with
`npm run build` locally, fix the error, and re-trigger — nothing was
published, so no version was consumed.

### npm rejects the publish as an existing version

The version in `package.json` was already published. Bump to the next
unused version and re-run; a consumed version number cannot be reused.
